1. Introduction

This document describes the CTP back office API, hereafter simply referenced as the BO-API. As such it complements the CTP API data model and API, hereafter simply referenced as the [client_API]. Concepts and reference material defined in the [client_API] will not be repeated here.

There are two key differences between the BO-API and the [client_API] that are worth noting. First — with few exceptions — the [client_API] mainly describes read-only API calls with the GET method, while the BO-API contains many write calls with the POST or PUT methods, mirroring the read-only methods in the client API. With an understanding of the [client_API] it should be straightforward to understand related calls in the BO-API. Second, the BO-API contains specific administrative API methods, for the purpose of configuring access control, through the creation of credentials (i.e. access tokens) and the definition of access rights (through the use of tags).

1.1. Notations

1.1.1. Identifiers

As described in the [client_API], CTP API method URLs are composed of two parts:

  1. A CTP entry point denoted {ctpBase}.

  2. A relative path to the entry point that represents a CTP API resource, accessible through specific HTTP methods.

The relative URL is itself further broken down into resource types (e.g. assets in /assets/a1b2c3) and resource identifiers (e.g. a1b2c3 in /assets/a1b2c3).

Example 1. Breaking down a CTP URL

The URL https://ctp.ikialab.net:4443/ctp/assets/a1b2c3 is broken down into:

  • {ctpBase}=https://ctp.ikialab.net:4443/ctp, and

  • the relative URL path /assets/a1b2c3.

The relative URL path is further broken down into a resource type “asset” and a resource identifier “a1b2c3”.

See the [client_API] for details.

1.1.2. API query signatures

In this document, we will often describe API calls through their signature.

Given an HTTP query to a fully qualified URL, a CTP API query signature S is created with the following algorithm:

STEP 1

Let M represent the HTTP method in uppercase (i.e. GET, POST, PUT or DELETE).

STEP 2

Let P represent the relative URL path to {ctpBase}, not including the query string (e.g. http://example.com/ctp/asset/01?x=tag → /asset/01)

STEP 3

Replace all resource identifiers in P with the symbol {id} (e.g. /assets/a1b2c3 → /assets/{id}).

STEP 4

If the query string does not contain a variable named x, go to step 6.

STEP 5

Append the character ? to P followed by the value of the query string variable x (e.g. http://example.com/ctp/asset/01?x=tag → /assets/{id}?tag)

STEP 6

Return S as the string concatenation of M, a space (U+0020) and P.

Example 2. A signature of a URL

A GET query to the URL http://example.com/ctp/assets/01?x=foo&y=bar will have the following signature:

GET /assets/{id}?foo

2. Access control

Access control in the BO-API is controlled by two elements:

  • Tokens: unpredictable random strings that serves to identify and authenticate principals (encoded as Base64 URL-safe strings [RFC_4648]).

  • Tags: textual strings that can be associated to tokens, API query signatures or API resources, which represent access rights.

Authentication is controlled by the verification, creation and deletion of tokens.

Authorisation is controlled by tags. Tags are associated with tokens, API resources and API query signatures. In order to determine if a principal is authorised to access an API resource, the server will perform tag matching verifications between the tags associated with the client’s token and the tag of the API resource as well as the tag of the API query signature as detailed hereafter.

2.1. Authentication

As described in the [client_API] specification, making a CTP API call requires the user to specify an authentication token, as shown in the example below using curl.

curl -u "token:qSWzVMzJ4Ee2nAKLWL5EWbnt" -X GET https://ctp.example.com/asset/01

With the BO-API, administrators can create tokens and associate tags to them.

This is done by issuing a POST request:

POST /tokens

This request must have a JSON body describing the tags associated with the newly created token, along with an optional annotation:

{
    "annotation": "this token is created for John Smith",
    "accessTags": [
        "access:user",
	    "access:anybody",
        "view:1234"
    ],
}

The server will acknowledge the creation of the new token with a 201 status code followed by a copy of the newly created resource, which will include a self property pointing to the token’s newly created URL along with a randomly generated token key:

{
    "self": "https://ctp.example.com/tokens/2931/",
    "annotation": "this token is created for John Smith",
    "accessTags": [
        "access:user",
	    "access:anybody",
        "view:1234"
    ],
    "key": "qSWzVMzJ4Ee2nAKLWL5EWbnt"
}
For reasons detailed at the end of the section on creating views, it is recommended to first create a view and then next create a token for the user associated with the view.

The URL returned in the self property of the response to the POST request can be used to later read the token, e.g.:

GET /tokens/2931

The same URL can also be used to delete the token, e.g.:

DELETE /tokens/2931
There is currently no way to modify a token with a PUT request. Modifications are done by deleting the token and reposting a new version of it.

Finally, the list of all existing tokens is accessible to administrators as a collection by issuing a GET request, e.g.:

GET /tokens/

2.2. Authorisation

2.2.1. Principles

Authorisation is based on tag matching.

Tags can be any UTF-8 string. By convention they are typically composed of two parts separated by a column (e.g. view:1234).

Tags are associated with 3 kinds of objects:

  1. Tokens (i.e. “users”) as described in Authentication.

  2. API query signatures, as described in API query signatures.

  3. API resources, such as an “asset”, a “view”, an “attribute”, etc.

In CTP there are two types of resources accessible by the API: individual resources and collections. For individual resources, access is granted by matching tags from Tokens with tags from both API query signatures and API resources. For collections, access is granted granted by matching tags from Tokens with tags from both API query signatures and the tags of the API resource that represents the scope of the collection.

More precisely:

  • a token (i.e. a “user”) will be authorised to perform a CTP API query to an individual resource if:

    • at least one tag associated with the token matches a tag associated with the API query signature.

    • at least one tag associated with the token matches a tag associated with the API resource.

  • a token (i.e. a “user”) will be authorised to perform a CTP API query to a collection:

    • at least one tag associated with the token matches a tag associated with the API query signature.

    • at least one tag associated with the token matches a tag associated with the API resource representing the scope of the resource (i.e the ``parent'' of the collection items in the CTP resource hierarchy):

      • EEE

      • This rescriction does not apply to collection of views.

Two tags are considered as matching if either:

  • they are strictly equal, OR

  • one of the tags is the wildcard value * (U+0042).

Use the wildcard value with care, associating it only to tokens destined to administrator. The wildcard allows in effect to override all access control restrictions.
Example 3. Matching tags

Consider a token with the following tags:

  • access:user

  • view:1234

A user want to use this token to access the following CTP URL with a GET request to https://ctp.example.com/assets/a1b2c3/. Such a request has the following API signature: GET /assets/{id} with {ctpBase}=https://ctp.example.com.

Further assume that:

  • The CTP server associates the tag access:user to the API signature GET /assets/{id},

  • The CTP server associates the tag view:1234 with the asset numbered a1b2c3.

The client will be allowed to access the resource because one tag in the token matches a tag associated to the API signature and another matches the resource itself. If the tag associated with the asset numbered a1b2c3 was view:6789 instead for example, access would be refused.

2.2.3. Tags of resources

An arbitrary set of tags can be associated with a resource by using the tag related methods described in this specification in API calls. These methods are easily distinguished from others because they have the query string parameter x=tags in their URL (i.e. their signature ends with ?tags).

When an CTP resource is created it is initiated with a set of tags that are chosen according to the following rules:

  • If the resource is a view, it is assigned a tag of the form view:{id}, where {id} is the unique identifier of the newly created view. This is the same {id} that is found in the URL of the view.

  • If the resource is a metric, it is assigned a set of tags containing only the predefined tag access:anybody.

  • In all other cases, the resources will inherit all the tags of its parents, namely:

    • A service-unit gets all the tags of view it is part of.

    • An asset gets all of the tags of the service-unit it is part of.

    • An attribute gets all of the tags of the asset it applies to.

    • A measurement gets all of the tags of the attribute it applies to.

    • A trigger gets all the tags of the measurement it applies to (note: unimplemented).

3. Resources

The JSON resources used in the BO-API are the same as in the [client_API] with the addition of two elements: tags and tokens. These two additional elements are detailed here.

3.1. Tag resources

Tag resources are meant to be consulted or modified by administrators. If /path/to/resource is a link to a CTP resource, then the corresponding tag resource can be accessed simply by appending the query string x=tags to the URL (e.g. /path/to/resource?x=tags). This does not apply to collections however.

Collections have no tags. Therefore in order to access a collection, access control tags matching is limited to matching the tag in the token with the tag of the API query signature.

The format of a tag resource is described hereafter.

{
  "self": URL,
  "accessTags": [
    string,
    ...
  ]
}
Property Type Description

self

URL

Self-link that can be queried to get an updated representation of the resource. It does not appear in request bodies, only in responses.

accessTags

list

List of strings that represent tags associated to the resource.

accessTags[]

string

A string representing a tag associated with the resource.

3.2. Token resources

Token resources describe user credentials. The credentials are represented as a random unpredictable string in the Base64 URL-safe alphabet [RFC_4648]. Administrators can read, create or delete token resources with the relevant methods described in this specification.

The format of a token resource is described hereafter.

{
  "self": URL,
  "accessTags": [
    string,
    ...
  ]
  "annotation": string,
  "key": string
}
Property Type Description

self

URL

Self-link that can be queried to get an updated representation of the resource. It does not appear in request bodies, only in responses.

annotation

string

An optional string representing textual information associated with the token.

accessTags

list

List of strings that represent tags associated to the resource.

accessTags[]

string

A string representing a tag associated with the resource.

key

string

A Base64 URL-safe string of 32 characters (representing 24 bytes of random value). It does not appear in request bodies, only in responses.

4.1. GET /views/{id}?tags

Get the access tags associated with a view.

Method characteristics

  • Request body: none

  • Query string: x=tag

  • Response body: a tag resource

  • Success status code: 200

Request example
GET /views/0123?x=tags

There is no request body.

Response example
{
  "self": "https://ctp.ikialab.net:8080/ctp/views/VIYQUT1WG628fhbQ?x=tags",
  "accessTags": [
    "view:VIYQUT1WG628fhbQ"
  ]
}

4.2. PUT /views/{id}?tags

Modify the access tags associated with a view.

Method characteristics

  • Request body: a tag resource, excluding the self property

  • Query string: x=tag

  • Response body: a tag tag resource.

  • Success status code: 200

Request example
PUT /views/0123?x=tags

The request body contains a tag resource.

{
  "accessTags": [
    "view:VIYQUT1WG628fhbQ",
    "foo:bar"
  ]
}
Response example

The response body is the same as the request body with the addition of a self-reference.

{
  "self": "https://ctp.ikialab.net:8080/ctp/views/VIYQUT1WG628fhbQ?x=tags",
  "accessTags": [
    "view:VIYQUT1WG628fhbQ",
    "foo:bar"
  ]
}

4.20. PUT /measurements/{id}?result

Set or update the measurement result within a measurement.

Method characteristics

  • Request body: the sub-part of a measurement defined as the content of the result property (see [client_API], section 4.2.6).

  • Query string: x=result

  • Response body: a measurement resource (see [client_API]).

  • Success status code: 200

The following properties may be omitted in the request body: dateTime, authority and signature. When omitted dateTime is set to the current date, while authority and signature are set to null.

Request example
PUT /measurements/VYkcv8y-5zuVAAAH?x=result

The request body corresponds to the content of the result property of a measurement resource (see [client_API], section 4.2.6).

{
  "value": [
     {
        "knots": 1
     }
  ],
  "dateTime": "2015-06-23T11:45:51.754+03:00",
  "authority": "net.ikialab",
  "signature": "eyJhbGciOiJSUz..." // shortened for clarity
}
Response example

The response is a measurement resource as defined in the [client_API] (section 4.2.6).

{
  "self": "https://ctp.ikialab.net:4443/ctp/measurements/VYkcv8y-5zuVAAAH/",
  "attribute": "https://ctp.ikialab.net:4443/ctp/attributes/VYkcv8y-5zuVAAAF/",
  "annotation": "",
  "metric": "https://ctp.ikialab.net:4443/ctp/metrics/VYkcv8y-5zuVAAAG/",
  "result": {
    "value": [
      {
        "knots": 1
      }
    ],
    "dateTime": "2015-06-23T11:45:51.754+03:00",
    "authority": "net.ikialab",
    "signature": "eyJhbGciOiJSUz..." // shortened for clarity
  },
  "objective": {
    "condition": "value[0].knots>5",
    "status": "false"
  },
  "createTrigger": null,
  "userActivated": false,
  "state": "activated"
}

4.22. POST /views

Create a new view.

Method characteristics

  • Request body: a view resource, with only the following property: annotation (see [client_API]).

  • Query string: none

  • Response body: a view resource, with all properties defined.

  • Success status code: 201

The properties serviceUnits, logs and triggers are omitted in the view posted in the query. They are automatically set by the server.

When a view is created, it is automatically assigned a tag of the form view:{id}, where {id} is the unique identifier assigned to the newly created view. This {id} is the same one that appears in the self property that is returned in the newly created view. This automatically assigned tag can be consulted or modified with the tag API.

Request example
POST /views/

The request body is a subset of the view resource defined in the main API.

{
   "annotation": "this is a view"
}
Response example
{
  "self": "https://ctp.ikialab.net:4443/ctp/views/VYkcvsy-5zuVAAAB/",
  "annotation": "this is a view",
  "serviceUnits": "https://ctp.ikialab.net:4443/ctp/views/VYkcvsy-5zuVAAAB/serviceUnits/",
  "logs": "https://ctp.ikialab.net:4443/ctp/views/VYkcvsy-5zuVAAAB/logs/",
  "triggers": "https://ctp.ikialab.net:4443/ctp/views/VYkcvsy-5zuVAAAB/triggers/"
}

The simplest approach to create a new user (i.e. a new token), is therefore to proceed as follows:

  1. Create a view (i.e. POST /views/).

  2. Get the tag view:{id} automatically associated with the newly created view (i.e. GET /views/{id}?x=tags).

  3. Create a token and use view:{id} from step 2 as one of the tags of the token (i.e. POST /tokens).

4.23. POST /views/{id}/serviceUnits

Creates a service-unit within the view identified by {id}.

The newly created service-unit automatically inherits all tags from the parent view.

Method characteristics

  • Request body: A service-unit resource as defined in the [client_API] (section 4.2.2), excluding the following properties: self, dependencies and assets, which are set by the server.

  • Query string: none

  • Response body: A service-unit resource.

  • Success status code: 201

The property serviceClass may also be omitted in the query. It will set set to the empty string by the server in that case.

Request example
POST /views/VYkcvsy-5zuVAAAB/serviceUnits

The request body is a serviceUnit resource as defined in the [client_API] (section 4.2.2) excluding the self property.

{
  "view": "https://ctp.ikialab.net:4443/ctp/views/VYkcvsy-5zuVAAAB/",
  "annotation": "this is a test to add a service.",
  "provider": "net.ikialab",
  "serviceClass": ""
}
Response example

The response body is a newly created serviceUnit, which includes a self property with the URL of the newly created object.

{
